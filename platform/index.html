平台管理方

用网络治理和规则治理，支撑承诺兑现。

平台管理方的职责不是逐票操作，而是定义哪些站点进入网络、哪些节点强控制、哪些接口优先接入， 并确保主数据、服务等级、异常字典和可信预留从第一天就不跑偏。

治理

平台后台要先管规则，再管菜单。

商业模型说明书反复强调：PRD 不能从页面出发，而要从节点、状态机、主键和异常字典出发。

主数据

平台必须冻结主键体系。

没有统一对象，后续所有接口、对账和可信映射都会变形。平台后台应单独提供对象治理视图。

服务规则

服务等级与硬门槛

P1 / 高优先级

仓、港、车三个关键环节必须体现优先级，不能只显示在订单字段里。

P2 / 标准级

纳入统一节点、统一异常和统一 KPI 体系，是网络复制的基准层。

节点硬门槛

未放行、未装机前复核、未二次复核、未完成 POD 的记录不得进入下一状态。

异常字典

数量、单证/标签、板位/托盘、时效、运输、质量六类异常必须结构化治理。

接口

接口治理要从第一天开始预留未来。

当前不把自己定义为 Crypto 公司，但要从第一版数据模型开始预留可信事件和审计链。

ERP 与 OS

回连合同、客户、航班、发票、结算与 SLA 引用，让履约结果回到经营事实。

供应链系统与 OS

统一 Order、Shipment、Warehouse、Truck、POD、Delivery 等对象映射。

航班状态接口

接入离场、起飞、预计到达、延误、备降、落地等事件，让飞行段不再是黑箱。

末端回传接口

接入派送单、承运商、派送中、已送达、失败原因码等状态。

POD / 签收接口

保证责任闭环和对账基础，把签收对象结构化，而不是只存一张图片。

可信事件预留

预留事件哈希、签名引用、存证引用字段，并不等于当前就上链。
