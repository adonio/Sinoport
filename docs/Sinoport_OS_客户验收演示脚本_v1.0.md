# Sinoport OS 客户验收演示脚本 v1.0

## 1. 演示目标

本次客户验收只讲一条完整故事线：

1. 平台端能看见全网站点态势、风险、规则和报表。
2. 货站端能围绕 `MME` 站点完成进港、出港、文档、任务、异常和审计闭环。
3. 移动端与后台读取同一批真实数据库对象，现场作业和办公室编排一致。
4. `Copilot` 能围绕真实对象提供上下文查询，不再只是演示壳。

本次主演示对象固定为：

- 进港航班：`SE803`
- 出港航班：`SE913`
- 进港 AWB：`436-10358585`
- Shipment：`in-436-10358585`
- 进港异常：`EXP-0408-001`

## 2. 推荐演示顺序

### Step 1：平台总览

入口：

- `/platform/operations`
- `/platform/stations`

要讲：

1. 先说明平台能同时看见主演示站 `MME`、健康对照站 `URC`、风险对照站 `RZE`。
2. 强调这不是单站 demo，而是平台对多站点、不同控制层级和不同阶段的统一可视化。
3. 说明当前风险、待处理动作和关键事件都能回到真实对象。

客户应看到：

- 有明显的健康站、风险站、接入中站点对比
- `MME` 是主演示站
- 平台首页指标不是单一绿色样板

验收点：

- 平台能区分不同站点状态
- 平台数字能解释到具体对象或链路

### Step 2：平台网络、规则、主数据

入口：

- `/platform/network`
- `/platform/rules`
- `/platform/master-data`
- `/platform/reports`

要讲：

1. `URC -> MME` 是当前强控制主链路。
2. `URC -> MST -> RZE` 是接入中、风险更高的协同链路。
3. 规则页说明哪些关键文件、任务和证据会真正阻断下游放行。
4. 主数据页说明对象治理不是孤立配置，而是会影响链路、任务和报表。

客户应看到：

- 至少一条 active lane
- 至少一条 onboarding / 风险 lane
- 规则、主数据、报表之间能形成业务口径闭环

验收点：

- 平台端不是只看静态字典，而是能讲清链路规则和治理对象

### Step 3：进入 `MME` 站点

入口：

- `/station/dashboard`

要讲：

1. `MME` 是当前样板站，进港和出港都在同一个站点完成验收。
2. dashboard 上的待处理动作、风险、文档、任务和异常都能继续深入。

客户应看到：

- 有进港、出港、文档、任务、异常的统一入口
- 待处理动作与下游对象一致

验收点：

- 货站总览不是单纯看板，能继续钻取真实业务对象

### Step 4：进港主线 `SE803`

入口：

- `/station/inbound`
- `/station/inbound/flights/SE803`
- `/station/inbound/waybills/436-10358585`
- `/station/shipments/in-436-10358585`

要讲：

1. `SE803` 已落地，当前已经进入进港理货和尾程准备。
2. `436-10358585` 是这条进港链的主样板 AWB。
3. Shipment 详情能串起航班、AWB、文档、任务、异常和审计。

客户应看到：

- 航班详情里有文档、任务、异常摘要
- AWB / Shipment 详情与 `SE803` 一致
- 同一个对象在多个页面里字段一致

验收点：

- flight / awb / shipment 形成对象闭环
- 不是三个孤立页面

### Step 5：文档、任务、异常闭环

入口：

- `/station/documents`
- `/station/documents/noa`
- `/station/documents/pod`
- `/station/tasks`
- `/station/exceptions`
- `/station/exceptions/EXP-0408-001`

要讲：

1. 当前 `EXP-0408-001` 代表一条真实的进港阻断链：理货复核未完成，NOA 放行被挂起。
2. 文档状态、任务状态、异常状态和对象审计会互相回连。
3. 这条链说明系统不只是“看见异常”，而是能讲清它为何阻断、由谁处理、怎样恢复。

客户应看到：

- NOA / POD 页有真实对象回连
- 任务列表与异常详情能互相跳转
- 对象审计有连续记录

验收点：

- 至少一条“发现异常 -> 挂起 -> 补动作 -> 恢复”的完整业务叙事

### Step 6：出港主线 `SE913`

入口：

- `/station/outbound`
- `/station/outbound/flights`
- `/station/outbound/flights/SE913`
- `/station/outbound/waybills`

要讲：

1. `SE913` 代表一条正在推进中的出港链。
2. 当前不是“完全正常”，而是故意保留 manifest 最终冻结前的阻断状态，方便验收关键门槛。
3. 出港页面能看到收货、装机复核、manifest 和 loaded / airborne 前的关键动作。

客户应看到：

- `SE913` 有正在处理中的动作
- 文档、任务、异常仍然会回到同一个出港对象链

验收点：

- 出港链不是只展示航班号，而是能讲清门槛和当前阻断原因

### Step 7：移动端执行

入口：

- `/mobile/select`
- `/mobile/inbound/SE803/breakdown`
- `/mobile/inbound/SE803/loading`
- `/mobile/outbound/SE913/receipt`
- `/mobile/runtime/SE913`

要讲：

1. 移动端和后台共用同一批对象，不靠浏览器本地假数据驱动作业。
2. `SE803` 进港链能演示拆板、点货、装车计划。
3. `SE913` 出港链能演示收货扫描、runtime 与装机准备。

客户应看到：

- 节点之间有连续数据
- 不会出现空页、白屏、断链
- PDA 页和后台详情页能对上同一航班 / AWB

验收点：

- 现场执行与后台编排读取同一批对象

### Step 8：Copilot

入口：

- `/station/copilot?object_type=Flight&object_key=SE803`
- `/station/copilot?object_type=AWB&object_key=436-10358585`
- `/station/copilot?object_type=Shipment&object_key=in-436-10358585`
- `/station/copilot?object_type=Exception&object_key=EXP-0408-001`

要讲：

1. Copilot 不是独立 demo，对象上下文直接来自真实数据库对象。
2. 以 `SE803`、`436-10358585`、`in-436-10358585`、`EXP-0408-001` 为例，能查到摘要、任务、文档、异常和审计信息。

客户应看到：

- 左侧会话列表
- 中间消息流
- 右侧工具与上下文
- 对象切换后上下文同步变化

验收点：

- Agent 能对真实对象回答“查得到”的问题

### Step 9：报表与收口

入口：

- `/station/reports`
- `/platform/reports`
- `/platform/audit`

要讲：

1. 平台报表负责讲多站点对比和治理口径。
2. 站点报表负责讲 `MME` 的当班执行结果和待处理动作。
3. 审计页证明整个演示链不是页面拼装，而是对象事件可回溯。

客户应看到：

- 报表和对象明细口径一致
- 审计里能看见主演示对象

验收点：

- 最终能把平台治理、货站操作、移动执行和审计回到一条统一主线

## 3. 客户常见问题答法

### Q1：这些数字是怎么来的？

答：

- 平台和货站的主要指标都来自真实对象聚合。
- 本次演示主对象固定为 `SE803 / SE913 / 436-10358585 / EXP-0408-001`，可以一路回到明细页解释。

### Q2：异常是怎么闭环的？

答：

- 异常会关联任务、文档和审计记录。
- 以 `EXP-0408-001` 为例，当前先展示阻断，再展示恢复动作应该如何推进。

### Q3：移动端和后台是不是两套数据？

答：

- 不是。
- 移动端和后台都从同一套数据库对象读取，区别只在操作界面和角色视图。

### Q4：Copilot 读的是什么？

答：

- 读的是对象上下文，不是单独的一份演示数据。
- 所以它能回到 flight / awb / shipment / exception 的真实摘要、任务、文档和审计链。

## 4. 收尾话术

最后只讲三件事：

1. 这套系统已经能把平台治理、货站执行、移动作业和智能查询放到同一条对象主线上。
2. 本次验收不是单页看图，而是基于真实数据库对象做连续讲解。
3. 当前样板站是 `MME`，后续复制到其它站点时，复用的是同一套对象模型和运行口径。
