# Sinoport 企业官网规划规范 v1

## 1. 文档目的

本文件用于把 Sinoport 企业官网从“概念性页面规划”进一步细化为一份可执行的官网规范文档，供以下角色共同使用：

- 合伙人：确认官网对外表达是否与商业模型一致
- 产品负责人：确认官网信息架构、页面范围与优先级
- 设计团队：确认页面层级、内容节奏、品牌表达和视觉重点
- 开发团队：确认一期官网的页面范围、模块结构和基础技术边界
- 运营团队：确认后续内容更新、案例维护、机场网络更新和 CTA 接收机制

本文件只讨论企业官网，不讨论后台管理页面的交互细节。

## 2. 官网定位

### 2.1 官网不是做什么

企业官网不应被做成以下任何一种形态：

- 传统物流公司宣传页
- 纯 SaaS 软件产品官网
- 大而全供应链门户
- 后台系统入口集合页
- 以 Crypto、RWA 或可信层为核心叙事的技术展示页

### 2.2 官网要做什么

官网要完成三件事：

1. 用一句话讲清 Sinoport 的产品定义
2. 用结构化页面讲清端到端履约控制逻辑
3. 把客户、合作站点、渠道伙伴引导到明确的下一步动作

### 2.3 官网的一句话定义

建议官网统一使用以下表达：

> Sinoport 不是在搬运货物，而是在兑现客户承诺。

英文对外统一表达建议：

> Promise-Driven Fulfillment Infrastructure

## 3. 官网目标

### 3.1 业务目标

- 让目标客户快速理解 Sinoport 卖的是“承诺兑现型履约”，不是单点 handling
- 让合作方理解 Sinoport 有可复制的节点控制逻辑和网络方法论
- 让潜在客户对网络、时效、透明度和责任闭环形成信任
- 让感兴趣对象完成咨询、合作申请或项目对接

### 3.2 转化目标

官网一期应至少承接三类转化：

- 业务咨询：客户提交需求，进入商务跟进
- 网络合作：合作货站、卡车公司、区域协同方提交合作意向
- 项目对接：重点客户、包机组织者、平台型客户发起专项沟通

### 3.3 非目标

官网一期不应承担以下任务：

- 展示后台系统所有功能细节
- 展示站内操作 SOP 全部细节
- 作为内部员工操作入口
- 过度介绍可信层、链上化、Tokenization 或 RWA
- 承担复杂客户门户功能

## 4. 目标受众

官网的主要读者不是同一类人，因此页面需要兼顾“快速理解”和“深入判断”两种需求。

### 4.1 需求端客户

典型对象：

- 跨境电商卖家
- 品牌客户
- 平台型客户
- 高时效供应链客户

他们最关心：

- 是否准时
- 是否透明
- 是否能定位异常与责任
- 是否覆盖关键节点

### 4.2 渠道与协同伙伴

典型对象：

- 货代
- 集货商
- 包机组织者
- 卡车公司
- 区域协同货站

他们最关心：

- Sinoport 的网络规则和接入方式
- 站点进入网络后的角色与要求
- 是否有统一 SLA、统一字段和统一责任链

### 4.3 合伙人与潜在合作机构

他们最关心：

- Sinoport 的产品边界是否清晰
- 官网表达是否与商业模型一致
- 网络扩展逻辑是否清楚
- 是否具备标准化复制潜力

## 5. 官网总原则

### 5.1 先讲产品定义，再讲功能

首页必须先回答：

- Sinoport 是什么
- 为什么不是传统货运叙事
- 为什么它解决的是“承诺兑现”问题

而不是一上来展示模块列表或系统截图。

### 5.2 先讲结果，再讲过程

官网不应把“收货、拆板、装车、派送”逐项堆成功能说明，而应先用结果语言表达：

- 准时
- 准确
- 透明
- 可追责

流程说明只服务于解释这个结果是如何做到的。

### 5.3 先讲网络和控制层，再讲单站点能力

客户购买的不是某一个站点的处理动作，而是网络中一组关键节点被统一控制后的确定性结果。

### 5.4 后台系统不应成为官网主角

Sinoport OS 应作为“控制层”被解释，而不是作为“软件产品截图集合”来展示。

## 6. 官网信息架构

建议一期官网采用以下一级结构：

1. 首页 `Home`
2. 产品与解决方案 `Promise-Driven Fulfillment`
3. 履约网络 `Network`
4. 控制系统 `Sinoport OS`
5. 应用场景 `Scenarios`
6. 关于与联系 `About / Contact`

### 6.1 不建议放入主导航的内容

以下内容不建议放入官网主导航：

- 平台管理方后台
- 货站管理员后台
- 内部资料中心
- 可信数据层深度说明

如果确实需要登录入口，建议只放在页脚或右上角低优先级位置，不作为官网主导航主项。

## 7. 首页规范

首页是整个官网最重要的页面，不追求信息最全，而追求“最短时间讲清楚 Sinoport 是什么”。

### 7.1 首页目标

首页必须完成以下判断链：

1. 让用户在 5 秒内知道 Sinoport 的定位
2. 让用户在 20 秒内理解端到端履约控制逻辑
3. 让用户在 60 秒内看到网络、节点、控制层和转化入口

### 7.2 首页模块顺序

建议按以下顺序排布：

1. Hero 首屏
2. Why Sinoport
3. 四个结果承诺
4. 五段端到端履约链路
5. 机场网络与节点布局
6. Sinoport OS 控制层说明
7. 适用场景
8. 合作与咨询 CTA

### 7.3 Hero 首屏

### 模块目标

用最少文字讲清产品定位和差异化。

### 必须出现的信息

- 主标题：承诺兑现型履约
- 核心口号：不是运货，而是兑现承诺
- 副文案：说明其覆盖端到端链路与关键节点控制
- 主要 CTA：预约业务沟通
- 次级 CTA：查看履约网络

### 建议文案方向

主标题建议：

`Promise-Driven Fulfillment for Cross-Border Air Logistics`

副标题建议：

`从中国集货仓到客户签收，以关键节点控制与数字化控制层交付准时、准确、透明、可追责的履约结果。`

### 7.4 Why Sinoport

### 模块目标

用对比方式快速帮助用户理解 Sinoport 的边界。

### 推荐表达结构

- 不是传统地服
- 不是单卖软件
- 不是项目制解决方案
- 不是大而全平台

### 页面形式建议

使用四个对照卡片，每张卡片只保留一句话，不写成长段说明。

### 7.5 四个结果承诺

该模块是官网的结果页心智核心，建议采用四个并列指标卡片：

- Timely
- Accurate
- Visible
- Accountable

每个指标都要有一句对客户有感知的话：

- Timely：在承诺时窗内完成关键节点与最终交付
- Accurate：件数、板位、AWB、卡车与签收信息一致
- Visible：关键状态、异常与责任链路实时可见
- Accountable：每个关键动作都能追溯到人、时间与来源

### 7.6 五段端到端履约链路

### 模块目标

让用户理解 Sinoport 不是单一机场工具，而是覆盖完整承诺兑现链路。

### 五段链路必须按以下顺序展示

1. 中国集货仓收货及前置履约
2. 出港机场履约
3. 飞行在途状态
4. 进港机场履约
5. 客户签收 / 最后一公里交付

### 每段链路的表达要求

每一段只保留三层信息：

- 这一段叫什么
- 这一段控制什么
- 这一段如何影响承诺兑现

不要在首页展开太多字段、状态和 SOP 细节。

### 7.7 机场网络与节点布局

### 模块目标

展示 Sinoport 已经覆盖或正在规划的机场节点网络，让用户看到“网络化”而不是“单站点”。

### 首页必须体现的机场

- `URC`
- `KGF`
- `NVI`
- `RZE`
- `MST`
- `BoH`
- `MME`

### 展示建议

不要只做代码列表，建议至少分出区域层次：

- 中国
- 中亚
- 欧洲
- 英国

### 额外要求

- `MME` 要明确标注为样板站 / SOP 样板站
- 页面上要体现不同节点存在不同控制深度

### 7.8 Sinoport OS 控制层说明

### 模块目标

解释系统是“控制层”，不是单独售卖的软件。

### 页面必须讲清四层架构

- ERP 底座层
- 供应链系统层
- Sinoport OS 控制层
- 可信数据层

### 官网表达边界

可信数据层只需要说明“已预留，不是当前主营”，不展开资产化叙事。

### 7.9 应用场景

首页建议只展示 3 个代表性场景，不要做得太散。

推荐场景：

1. 跨境电商航空履约
2. 高时效品牌补货
3. 区域网络协同中转

每个场景用一句话说明：

- 为什么该场景需要承诺兑现型履约
- Sinoport 在该场景中的价值点是什么

### 7.10 首页 CTA

首页至少配置 3 类 CTA：

- 预约业务沟通
- 申请网络合作
- 获取试点方案

CTA 不要写得太泛，建议按钮文案直接表达下一步动作。

## 8. 二级页面规范

### 8.1 产品与解决方案页

### 页面目标

从业务逻辑角度展开说明 Sinoport 的产品结构，而不是列系统功能。

### 推荐结构

1. 产品定义
2. 客户为什么购买
3. 五段履约链路
4. 服务等级与结果承诺
5. 与传统方案的区别
6. CTA

### 8.2 履约网络页

### 页面目标

把机场网络、区域布局、控制深度和样板站表达清楚。

### 推荐结构

1. 网络总览
2. 分区域机场节点
3. 控制深度说明 `L1 / L2 / L3`
4. 样板站 `MME`
5. 网络扩展与合作入口

### 8.3 Sinoport OS 页

### 页面目标

讲清控制层逻辑，但不把这一页做成产品后台说明书。

### 推荐结构

1. 系统为什么存在
2. 四层架构
3. 关键控制能力
4. 接口预留原则
5. 审计链与责任闭环

### 建议重点

这一页可以讲：

- 节点可视化
- SLA
- 优先级
- 异常归因
- POD 闭环
- 接口接入

这一页不建议讲：

- 所有字段表
- 所有后台菜单
- 链上技术细节

### 8.4 应用场景页

### 页面目标

帮助不同行业角色快速判断“我是否适合 Sinoport”。

### 推荐结构

按场景拆分：

- 跨境电商
- 品牌客户
- 平台型客户
- 区域协同网络

每个场景回答三个问题：

- 痛点是什么
- Sinoport 如何解决
- 下一步如何对接

### 8.5 关于与联系页

### 页面目标

强化可信度，并承接所有转化。

### 建议内容

- 公司简介
- 产品定位摘要
- 网络与区域概览
- 联系方式
- 咨询表单

## 9. 内容规范

### 9.1 文案语气

官网文案应体现以下特征：

- 明确
- 克制
- 结果导向
- 不说空话
- 不堆砌行业黑话

### 不建议的表达

- 一站式全链路赋能
- 智慧化数字化全栈平台
- 生态闭环引领行业升级
- 区块链重塑物流未来

### 建议的表达

- 兑现客户承诺
- 控制关键节点
- 打通履约真相与经营真相
- 把异常和责任结构化

### 9.2 视觉表达规则

官网视觉要体现“控制力”和“结果确定性”，而不是普通物流海报风。

建议方向：

- 版面结构清楚
- 节点与链路清晰
- 网络感强
- 不做嘈杂装饰
- 不依赖系统截图堆砌专业感

### 9.3 Logo 使用规则

官网应统一使用透明底 logo 资产，避免白底图片重复出现在不同背景中。

建议原则：

- 头部导航、页脚、分享图和资料页统一使用同一套 logo 资产
- 不在同一页面混用多种 logo 版本
- 如果页面背景较浅，优先使用标准彩色 logo
- 如果页面背景较深，可准备深色背景适配版本，但主字形和比例不变

## 10. 页面中必须保留的数据与事实

官网中涉及事实信息时，必须统一口径，不能每页写法不同。

### 必须统一的内容

- 公司名称：`Sinoport`
- 核心定位：`Promise-Driven Fulfillment`
- 四项结果承诺：`Timely / Accurate / Visible / Accountable`
- 五段履约链路
- 七个机场节点
- `MME` 样板站定位

### 不建议在官网一期过度公开的内容

- 详细 KPI 阈值
- 内部 SOP 具体作业细节
- 全部接口字段
- 审计链具体实现
- 可信层技术细节

## 11. 官网中的 CTA 与表单规范

### 11.1 表单类型

建议官网至少配置以下三种表单入口：

1. 业务咨询
2. 合作站点申请
3. 项目试点沟通

### 11.2 表单字段建议

业务咨询表单建议字段：

- 公司名称
- 联系人
- 联系方式
- 业务区域
- 需求描述

合作申请表单建议字段：

- 公司名称
- 合作类型
- 覆盖区域 / 机场
- 联系方式
- 简要能力说明

### 11.3 表单后的动作

表单提交后，官网必须至少做到：

- 明确提示“已收到”
- 说明预计响应时间
- 将线索按类型进入不同跟进流程

## 12. 一期官网范围建议

为了确保官网先规范后建设，建议一期范围控制在以下页面：

1. 首页
2. 产品与解决方案页
3. 履约网络页
4. Sinoport OS 页
5. 应用场景页
6. 关于与联系页

## 13. 一期官网不建议做的内容

- 多语言一次性做太多版本
- 动画过重的视觉表达
- 大量系统截图轮播
- 案例库过重
- 登录体系复杂化
- 后台入口做成主导航

## 14. 官网规划结论

Sinoport 企业官网的重点不是“展示一个物流公司”，而是“把承诺兑现型履约这件事讲明白”。

如果一句话概括官网规范重点，就是：

> 用结果价值建立认知，用端到端链路建立理解，用网络与控制层建立信任，用明确 CTA 承接转化。
