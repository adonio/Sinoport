## 2. 当前判断

### 2.1 已经具备的基础

- 信息架构已经从评审版升级到接近开发版的规模
- 平台端、货站端、PDA 端都已有较完整的前端入口
- 当前最有价值的资产，不是单页视觉，而是：
  - 菜单结构
  - 对象关系
  - 状态链表达
  - 任务 / 文件 / 审计的页面边界

### 2.2 当前最主要的问题

当前问题不再是“有没有页面”，而是“页面后面的系统骨架是否统一”。主要缺口集中在：

- 对象字典未冻结
- 字段字典未冻结
- 状态归属矩阵未冻结
- 文件驱动与任务驱动的统一机制还停留在 demo 表达
- 前后端 contract 还未正式定义

## 3. 后续开发总原则

1. 先统一模型，再写接口。
2. 先冻结状态，再做自动流转。
3. 先让 `MME` 样板站闭环，再复制到其他站点。
4. 先让前端和测试口径一致，再进入后端实现。
5. 所有“文件、任务、状态、审计”必须基于同一主键体系。

## 4. 建议分阶段推进

### Phase 1：开发冻结

目标：

- 把开发基线冻结，避免边开发边改概念。

必须完成：

- 冻结信息架构
- 冻结对象字典
- 冻结字段字典
- 冻结状态归属矩阵
- 冻结硬门槛规则
- 冻结文件链清单

本阶段输出物：

- 开发版 PRD
- 字段字典
- 状态归属矩阵
- 权限矩阵
- API Contract 草稿

### Phase 2：样板站闭环

目标：

- 先让 `MME` 进港样板站形成最小真实闭环。

优先模块：

- 进港管理
- 进港航班详情
- 提单与履约链路
- 单证与指令中心
- PDA 进港节点
- 异常中心

关键验收点：

- 一票货从到港到交付的状态闭环
- `NOA / POD / 二次转运` 都能挂到同一条对象链
- 异常能阻断、恢复和审计

### Phase 3：出港闭环

目标：

- 形成 URC 或标准出港链路的完整闭环。

优先模块：

- 出港管理
- 航班管理
- 提单管理
- 主单
- `FFM / UWS / Manifest`
- 机坪放行与飞走确认
- PDA 出港节点

关键验收点：

- 出港预报到 Manifest 的完整链路
- 目的港回传状态可对账
- 飞走前门槛能起作用

### Phase 4：平台化治理

目标：

- 从样板站能力升级到平台治理能力。

优先模块：

- 运行态势中心
- 主数据与接口治理
- 审计与可信留痕
- 平台级报表

关键验收点：

- 平台能看到跨站点、跨链路的统一视图
- 主数据不再散落在各页面
- 接口状态、失败重试、导入日志可管理

## 5. 具体开发意见

### 5.1 先做字段字典，不要直接开后端表

建议：

- 先从 `Flight / Shipment / AWB / ULD / Truck / Task / Document / Exception` 八类对象开始
- 每个字段必须补：
  - 类型
  - 是否必填
  - 是否人工可编辑
  - 来源
  - 是否参与状态推进
  - 是否进入审计

原因：

- 当前页面已经很多，如果字段字典不先定，后端表、接口和前端表单会很快失控。

### 5.2 状态归属矩阵必须早于接口开发

建议：

- 先用一张表把 `Flight Runtime / Ground Fulfillment / Task / Document` 四条状态链挂到具体对象上

原因：

- 没有这张表，测试无法写断言，后端无法定义写入点，前端也会持续重复表达。

### 5.3 单证与指令中心必须持续按正式口径推进

建议：

- 不要再把文件只当列表页
- 必须明确：
  - 文件触发什么任务
  - 文件缺失阻断什么状态
  - 文件替换是否需要复核

原因：

- 客户已经明确要求“文件驱动执行”，这是系统和普通后台的核心区别。

### 5.4 PDA 端应该按“任务卡”统一，不按页面复制

建议：

- 所有节点先统一成“任务卡 + 通用动作栏 + 证据区”
- 不要每个节点都独立长出一套完全不同的操作结构

原因：

- 这样后面扩展新站点和新节点的成本最低
- 测试和培训也更容易统一

### 5.5 样板站必须先闭环，不建议同步铺太多站点

建议：

- 先以 `MME` 做进港闭环
- 再选一条标准出港链路做出港闭环
- 之后再复制到别的站点

原因：

- 现在最缺的不是页面数量，而是“一条链能不能从头到尾跑通”

### 5.6 先设计接口 contract，再决定 mock 数据结构

建议：

- 当前 `data/sinoport.js` 很适合继续做 demo 数据
- 但下一阶段应新增一层 mapper / adapter
- 页面不要直接绑定未来后端返回格式

原因：

- 否则前端现有 demo 会在后端接入时整批返工

## 6. 优先级建议

### P0 必须先做

- 开发版 PRD 冻结
- 字段字典
- 状态归属矩阵
- `MME` 进港闭环
- 单证与指令中心升级
- 任务卡统一模型

### P1 紧接着做

- 出港闭环
- 提单与履约链路统一页
- 作业指令中心
- 资源管理
- 平台运行态势中心联动

### P2 后续补强

- 平台报表
- 可信留痕展示深化
- 多语言与多站点复制能力
- 更完整的角色与技能字典

## 7. 建议客户下一轮评审重点

建议客户下一轮不要再泛看整站，而是重点审这几件事：

1. 对象模型是否正确
2. 状态流是否正确
3. 文件链是否正确
4. PDA 操作是否符合现场习惯
5. 样板站闭环是否真实可执行

## 8. 结论

当前项目已经不缺“页面骨架”，下一阶段开发的关键不是继续堆页面，而是把：

- 对象
- 字段
- 状态
- 文件
- 任务
- 审计

统一到同一套系统语言里。

建议正式进入“开发冻结 + 样板站闭环 + 出港闭环 + 平台治理”的顺序，不建议再回到只围绕页面菜单做增删。
