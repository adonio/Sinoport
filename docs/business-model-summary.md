# Sinoport 商业模型摘要

## 1. 产品定义

Sinoport 的核心不是传统货运、地服外包或软件订阅，而是 `Promise-Driven Fulfillment`。

一句话定义：

> Sinoport 不是在搬运货物，而是在兑现客户承诺。

对外价值主张：

- 准时 `Timely`
- 准确 `Accurate`
- 透明 `Visible`
- 可追责 `Accountable`

## 2. 明确边界

Sinoport 不应被定义为以下任何一种单独形态：

- 传统地服公司
- 通用 SaaS 公司
- 项目制解决方案公司
- 大而全供应链平台
- 当前阶段的 Crypto / RWA 公司

系统的角色是控制中枢，不是独立售卖的软件。

## 3. 商业模型核心

### 谁付费

- 需求端客户：跨境电商卖家、品牌方、平台型客户、高时效供应链客户
- 渠道型客户：货代、集货商、包机组织者、物流整合商

### 主要收入层

- 基础 handling 与履约执行收入
- 履约等级溢价
- 承诺兑现溢价
- 网络协同收入
- 接口 / 数据增值收入

## 4. 运营方法论

关键判断不是“轻不轻资产”，而是“关键节点有没有控制权”。

建议控制原则：

- 关键仓、关键进港/出港站点、装车放行、异常处理要强控制
- 干线卡车可合作，但必须统一 SLA 与轨迹回传
- 飞行在途与最后一公里可先接口接入，但状态链不能断

## 5. 一期产品边界

Sinoport OS 一期必须覆盖端到端五段链路：

1. 中国集货仓收货及前置履约
2. 出港机场履约
3. 飞行在途状态
4. 进港机场履约
5. 客户仓签收 / 最后一公里交付

控制深度分为三层：

- `L1` 强控制节点
- `L2` 协同控制节点
- `L3` 接口可视节点

## 6. 四层数字架构

### ERP 底座层

- 客户、合同、包机项目
- 成本收入
- 发票与结算
- 经营分析

### 供应链系统层

- 订单
- 仓
- 车
- 交付
- 库存 / 分拨
- POD 与末端接入

### Sinoport OS 控制层

- 节点可视化
- SLA
- 优先级
- 异常归因
- POD 闭环
- 接口接入

### 可信数据层

- Event ID
- Event Hash
- Signature Reference
- Notarization Reference

当前阶段只做预留，不做资产化承诺。

## 7. 一期必须预留的接口

- `ERP ↔ OS`
- `供应链系统 ↔ OS`
- 航班状态接口
- 最后一公里接口
- `POD / 签收接口`
- 可信事件写入接口

## 8. 必须统一的主键

- `Order ID`
- `Shipment ID`
- `AWB / HAWB`
- `ULD / PMC ID`
- `Truck ID`
- `Flight ID`
- `POD ID`
- `Delivery ID`
- `Customer ID`
- `Contract ID / Invoice ID`
- `Event ID / Event Hash`

## 9. 节点建模要求

PRD 不能从“页面功能”开始，而要从“节点状态机”开始。每个节点至少要回答：

- 谁录入
- 录入什么
- 什么条件进入下一状态
- 异常如何分类
- 该节点如何影响 KPI 与责任链

## 10. MME 样板站的价值

说明书将 `MME` 作为最成熟的进港样板，重点 SOP 包括：

- 逐箱核对 `AWB`
- 相同提单号必须同托，严禁混托
- 拆板核对 `PMC`
- 卡车到达后二次复核
- 装车前登记司机 / 车牌 / 时间
- 装车双人操作
- `POD` 双签
- 落地后 `12` 小时内完成

这意味着后台系统应优先把 `MME SOP` 结构化为：

- 节点
- 字段
- 校验规则
- 异常码
- 角色权限
- KPI 口径

## 11. MVP 成功口径

MVP 不是做全，而是验证三件事：

- 端到端关键节点可见
- 关键节点真正可控
- 承诺兑现可量化、可追责

MVP 应优先保证：

- 节点状态更新
- 关键时间戳
- 异常标记与归因
- 优先级
- `POD` 闭环
- 航班状态接入
- 末端状态接入
- 审计链与 `Event Hash` 字段预留
