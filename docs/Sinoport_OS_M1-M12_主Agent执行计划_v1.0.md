## 3. 固定执行规则

### 3.1 主 agent 规则

主 agent 固定负责：

1. 读取总纲文档与现有台账
2. 派发当月任务给子 agent
3. 验收子 agent 输出
4. 通过后再进入下一月
5. 维护唯一月度执行文档

### 3.2 子 agent 规则

子 agent 固定负责：

1. 只细化当前月份
2. 输出月目标、周拆分、输入/输出、依赖风险、验收标准、决策点
3. 不扩展到下一月
4. 不做最终验收

### 3.3 串行规则

1. 同一时间只允许 `1` 个子 agent 处于实施态
2. 当前月份未验收通过前，不得进入下一月
3. 主 agent 认为“范围不清、验收不清、风险不清”时，必须要求子 agent 进一步细化
4. `M12` 验收通过后，才算本轮任务完成

## 4. 年度主线

未来 `12` 个月只围绕以下五条主线展开：

1. 生产安全收口
2. 构建产物与发布治理
3. 真实业务数据接入与单站试运行
4. 报表、多站点和运营稳定化
5. Agent 生产价值验证与年度收口

## 5. M1-M12 月度执行计划

### M1：生产安全收口

**月目标**

1. 把生产环境里的明文/占位安全配置全部收口到 Cloudflare/GitHub secrets。
2. 把 `demo-token`、`X-Debug-*`、`sinoport-local-dev-secret` 严格限制到本地开发。
3. 收紧 demo 账号与 bootstrap 逻辑。
4. 让 Web、API、Agent 三条认证口径一致。

**周拆分**

- `W1`：安全基线盘点与策略冻结
- `W2`：密钥与配置收口
- `W3`：生产鉴权收紧
- `W4`：验证、回归与发布护栏

**每周输入 / 输出**

- `W1`
  - 输入：auth/login/wrangler/前端登录链代码、生产与本地环境边界
  - 输出：安全基线表、fallback 边界表、风险清单
- `W2`
  - 输入：安全基线表、现有 Cloudflare/GitHub secrets 约定
  - 输出：secrets 收口方案、环境变量矩阵、配置变更表
- `W3`
  - 输入：secrets 方案、认证口径冻结结果
  - 输出：生产鉴权收紧补丁方案、本地/生产行为差异说明、受影响接口清单
- `W4`
  - 输入：收紧后的认证代码、CI/smoke 脚本
  - 输出：验收结果、回滚步骤、M1 完成判定

**依赖与风险**

- 依赖：Cloudflare secret 可写权限、GitHub secrets/environments 可更新
- 风险：误伤本地联调、旧脚本仍依赖 demo 账号、secret 虽迁移但运行时仍读取占位值

**验收标准**

1. production/staging 不再依赖 `demo-token`、`X-Debug-*`、`sinoport-local-dev-secret`
2. `AUTH_TOKEN_SECRET` 不再出现在生产明文配置里
3. demo 账号只允许 local 使用
4. `station / mobile / agent` 三端鉴权一致

**主 agent 决策点**

1. 是否保留 demo 账号作为 local-only fallback
2. secret 命名和环境拆分规则
3. `X-Debug-*` 是否彻底降级为 local-only 开发开关

### M2：构建产物治理

**月目标**

1. 把 `admin-assets/` 和根级静态 `index.html` 收敛为发布产物。
2. 消除“业务源码改动”和“构建产物改动”混在一起的问题。
3. 固化 `CI / 临时目录生成` 策略。
4. 明确产物是否继续入库，并写进仓库规范。

**周拆分**

- `W1`：现状盘点与策略冻结
- `W2`：发布产物边界方案设计
- `W3`：仓库治理规则固化
- `W4`：验收与回归口径

**每周输入 / 输出**

- `W1`
  - 输入：构建产物治理文档、当前 `git status`、发布脚本
  - 输出：构建产物清单、污染范围清单、策略草案
- `W2`
  - 输入：策略草案、发布链路、目录结构
  - 输出：边界方案、目录策略、是否入库建议
- `W3`
  - 输入：方案、README、任务表、发布脚本说明
  - 输出：治理规则文档、操作规范、回滚建议
- `W4`
  - 输入：治理规则、污染样态、CI/发布脚本
  - 输出：验收清单、回归口径、最终建议

**依赖与风险**

- 依赖：当前静态站构建链路、Pages 发布链路
- 风险：只改 `.gitignore` 不改发布流程，问题会反复出现

**验收标准**

1. `admin-assets/` 与根级静态 `index.html` 的归属明确
2. 发布流程能区分源码提交和产物提交
3. 主工作区的 `git status` 噪音问题有明确解决方案

**主 agent 决策点**

1. 最终选择“继续入库”还是“临时产物”
2. 是否把构建产物治理纳入发布前检查项

### M3：发布与运维基线

**月目标**

1. 固化 `staging / production` 的发布链路
2. 固定健康检查与版本校验
3. 把 D1/R2 备份、恢复、回滚写成运维基线
4. 把环境边界和密钥边界写清楚

**周拆分**

- `W1`：现状盘点与发布基线冻结
- `W2`：发布流程固化
- `W3`：运维与恢复基线
- `W4`：验收与发布护栏

**每周输入 / 输出**

- `W1`
  - 输入：wrangler 配置、GitHub Actions、Cloudflare 资源列表、现有运维文档
  - 输出：发布资产清单、健康检查清单、环境矩阵、风险清单
- `W2`
  - 输入：资产清单、现有 workflow/release 流程
  - 输出：标准发布流程、发布检查表、回滚入口说明
- `W3`
  - 输入：当前 D1/R2 使用方式、日志与审计现状
  - 输出：运维 Runbook、备份恢复手册、故障响应建议
- `W4`
  - 输入：发布流程与运维基线、staging/production 环境
  - 输出：验收报告、发布后检查结果、最终护栏建议

**依赖与风险**

- 依赖：Cloudflare staging/production 资源可访问
- 风险：配置不对称、发布流程未固化到 CI、版本与域名口径不统一

**验收标准**

1. 文档中有完整发布流程
2. staging/production 都有固定检查项
3. D1/R2 的备份恢复方式明确
4. 主 agent 可以据此判断一次发布是否真正完成

**主 agent 决策点**

1. 是否强制要求 staging 先验收再进 production
2. 发布后验收是否固定为 CI 必跑步骤

### M4：真实数据导入一期

**月目标**

1. 把 demo 过渡层和真实导入链统一到同一套对象模型
2. 把 `station-bundle-import` 和 `inbound bundle replay` 提升为正式一期导入链
3. 先把一条稳定导入链做牢，再扩大范围
4. 让导入链支持 demo 回放、真实导入、失败追踪与重放

**周拆分**

- `W1`：导入链盘点与口径冻结
- `W2`：正式导入链稳定化
- `W3`：真实数据接入与审计落库
- `W4`：验收、回放与试运行准备

**每周输入 / 输出**

- `W1`
  - 输入：`station-bundle-import`、fixture/replay、`demo_datasets`、对象链现状
  - 输出：导入口径表、对象范围清单、非目标清单
- `W2`
  - 输入：口径表、bundle import 与 replay 现状
  - 输出：导入流程说明、字段映射清单、失败语义说明、幂等键口径
- `W3`
  - 输入：正式导入链、真实业务数据样本
  - 输出：真实数据映射方案、审计落库规则、回放共用说明
- `W4`
  - 输入：导入链与审计链、真实数据导入结果
  - 输出：验收报告、回滚/重试说明、试运行准备清单

**依赖与风险**

- 依赖：现有导入链、replay 机制、对象链落库能力
- 风险：映射口径频繁变动、导入链缺少审计与回放、导入面铺太大

**验收标准**

1. 至少一条正式导入链可重复执行
2. demo 与真实数据进入同一对象链
3. 导入结果能写入审计并可回放
4. `request_id / Idempotency-Key` 具备正式幂等语义
5. 至少同时具备成功样本、重复样本、失败样本三类固定验收输入

**主 agent 决策点**

1. 第一批是否只接一条 inbound bundle
2. `demo_datasets` 是否严格限制为后端回放层
3. `request_id` 是否升级为正式幂等键

### M5：单站真实试运行

**月目标**

1. 选 `1` 个真实站点进入试运行
2. 把真实导入链、样板站对照组和 inbound 主链串成试运行闭环
3. 用真实数据验证对象链联动稳定性
4. 形成可复用的试运行 SOP 和问题回收机制

**周拆分**

- `W1`：试运行站点与边界冻结
- `W2`：试运行导入与回放准备
- `W3`：真实站点闭环联调
- `W4`：试运行验收与 SOP 固化

**每周输入 / 输出**

- `W1`
  - 输入：真实导入链、`MME` 样板站、真实环境可用性
  - 输出：站点清单、试运行边界表、排除项清单
- `W2`
  - 输入：站点主数据、导入链、数据样本
  - 输出：导入清单、映射表、回放/重试步骤
- `W3`
  - 输入：导入后的真实数据、inbound 主链、审计链
  - 输出：联调问题清单、闭环检查表、操作节奏建议
- `W4`
  - 输入：联调结果、试运行日志
  - 输出：验收报告、SOP 文档、问题回收列表

**依赖与风险**

- 依赖：真实导入链稳定、审计链可用、inbound 主链已完成基本收口
- 风险：试运行范围定义过大、导入链不稳定、站点操作节奏不一致

**验收标准**

1. 有且仅有 `1` 个真实站点进入试运行
2. inbound 主链能在真实站点上完整跑通
3. 问题能按对象、动作、字段定位

**主 agent 决策点**

1. 试运行站点选哪一个
2. 是否使用 staging 预演还是直接 production 试运行

### M6：数据质量治理

**月目标**

1. 把真实导入链、单站试运行和 `demo_datasets` 纳入同一套质量规则
2. 建立字段完整性、关联完整性、状态一致性、审计一致性的基线
3. 把问题发现推进到可重复的质量检查、分级、回放定位和归因
4. 让日报和对象链能反向暴露质量问题

**周拆分**

- `W1`：口径冻结与质量指标基线
- `W2`：质量检查规则与异常分类
- `W3`：质量追踪与回放定位
- `W4`：试运行复盘与治理固化

**每周输入 / 输出**

- `W1`
  - 输入：真实导入链、试运行数据、`demo_datasets`、审计/状态迁移、日报口径
  - 输出：指标表、分级表、检查范围表、角色边界说明
- `W2`
  - 输入：指标基线、样本数据、试运行日志
  - 输出：规则清单、异常字典、问题样本对照、处理模板
- `W3`
  - 输入：规则清单、审计链、日报与对象链
  - 输出：追踪流程、回放 SOP、责任归属建议、问题记录模板
- `W4`
  - 输入：试运行问题单、站点反馈、日报结果
  - 输出：复盘报告、治理手册、运营检查表、后续待办列表

**依赖与风险**

- 依赖：真实导入链、单站试运行、审计链、日报口径已具备基础能力
- 风险：规则定义过细、没有异常级别、日报与对象链口径不一致

**验收标准**

1. 覆盖核心对象的质量规则
2. 任一数据问题都能分类、定位、记录并给出处理动作
3. 日报、对象链、审计链形成反向追踪关系

**主 agent 决策点**

1. 是否把某些高严重级别问题设为默认阻断导入
2. `demo_datasets` 是否完全禁止前端直接消费

### M7：出港深化

**月目标**

1. 把 `loaded / manifest finalize / airborne` 从“可用”推进到“可稳定验收”
2. 把出港航班、提单、文档、任务、异常、日报、审计串成完整运营链
3. 让出港异常具备恢复、解除阻断和追责能力
4. 明确出港与进港的边界和联动规则

**周拆分**

- `W1`：出港口径冻结与链路盘点
- `W2`：出港动作链加固
- `W3`：出港对象联动与异常恢复
- `W4`：日报、验收与固化

**每周输入 / 输出**

- `W1`
  - 输入：已完成的出港动作链、对象链、日报现状
  - 输出：出港口径表、状态推进图、非目标清单
- `W2`
  - 输入：出港动作接口、对象模型、审计链
  - 输出：动作链规则说明、错误码表、审计约定
- `W3`
  - 输入：详情页、异常链、审计与日报
  - 输出：联动规则、恢复 SOP、详情页检查清单
- `W4`
  - 输入：动作链结果、联动结果、日报与审计
  - 输出：验收报告、日报检查项、待办清单

**依赖与风险**

- 依赖：出港基础能力与对象详情页已存在
- 风险：动作链失败语义不统一、异常恢复规则不明确、日报口径失真

**验收标准**

1. 三个动作的前置校验、失败语义、状态推进规则完整统一
2. 出港对象链、审计链、日报链可互相解释
3. 出港异常具备可恢复闭环

**主 agent 决策点**

1. 出港异常恢复是否允许自动重试
2. `manifest finalize` 之后是否允许修改对象

### M8：报表稳定化

**月目标**

1. 把平台日报、站点日报、异常日报统一到稳定口径
2. 把报表从“能展示”推进到“能运营”
3. 把数据质量治理结果反向接入报表
4. 为多站点运营提供可横向对比的日报基线

**周拆分**

- `W1`：报表口径冻结与现状盘点
- `W2`：日报计算与刷新机制稳定化
- `W3`：报表与数据质量联动
- `W4`：多站点对比、验收与固化

**每周输入 / 输出**

- `W1`
  - 输入：当前各类日报、对象链现状、数据质量治理结果
  - 输出：报表口径表、指标来源表、非目标指标清单
- `W2`
  - 输入：指标口径、对象链和审计链
  - 输出：刷新规则、重算与补算说明、追溯关系说明
- `W3`
  - 输入：数据质量规则、报表结果
  - 输出：异常提示规则、问题标记表、追踪路径说明
- `W4`
  - 输入：多站点数据、稳定化后的日报结果
  - 输出：多站点对比口径、检查清单、验收报告

**依赖与风险**

- 依赖：日报已有基础入口，对象链与审计链可追溯，多站点治理已具备初步能力
- 风险：口径漂移、覆盖面过大、质量问题未接入报表

**验收标准**

1. 三类日报核心口径统一且可追溯
2. 报表能稳定重算、补算、修正
3. 报表能反向暴露质量问题

**主 agent 决策点**

1. 哪些日报指标必须冻结为统一口径
2. 数据质量问题在报表里是仅提示还是阻断标记

### M9：多站点复制

**月目标**

1. 固化可复用的站点复制模板
2. 让新站点接入不再依赖人工逐页配置
3. 让多站点报表与平台治理能力对齐到同一套站点口径
4. 形成“新增站点接入清单 + 站点模板包 + 验收标准”的闭环

**周拆分**

- `W1`：站点复制口径冻结
- `W2`：站点复制包设计
- `W3`：多站点对齐与报表联动
- `W4`：验收、回放与固化

**每周输入 / 输出**

- `W1`
  - 输入：现有站点模板、主数据治理、配置模型、多站点报表现状
  - 输出：复制边界表、模板/覆盖项清单、最小接入单元定义
- `W2`
  - 输入：复制边界表、当前配置结构、治理页面/流程现状
  - 输出：复制包格式、校验规则、冲突处理规则、回滚规则
- `W3`
  - 输入：复制包设计、多站点报表口径、平台治理现状
  - 输出：对比指标口径、差异定位路径、站点接入检查项
- `W4`
  - 输入：复制流程与对齐结果、试跑样本、报表结果
  - 输出：验收报告、SOP 文档、问题回收列表、后续待办

**依赖与风险**

- 依赖：报表口径稳定、站点治理模板存在、配置模型可复用
- 风险：模板与覆盖项边界不清、站点特有配置过多、复制失败污染现有配置

**验收标准**

1. 至少有一套可复用的站点复制包
2. 新站点接入不再依赖逐页手工配置
3. 多站点报表与平台治理能按统一口径对比站点差异

**主 agent 决策点**

1. 哪些内容必须强制一致，哪些允许站点覆盖
2. 是否要求至少跑通第 `2` 个真实站点作为证明

## 6. 主 agent 月度验收规则

每个月统一按以下规则验收：

1. 月目标是否足够聚焦，没有扩散到其他月份
