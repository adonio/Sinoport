# Sinoport OS 升级项目开发任务规划

## 1. 文档定位

本文件基于以下三份事实同时制定：

- PRD 主依据：[Sinoport_OS_PRD_v1.0_升级开发版.md](/Users/lijun/Downloads/Sinoport/Sinoport_OS_PRD_v1.0_升级开发版.md)
- 现有原型依据：[platform-and-station-prd.md](/Users/lijun/Downloads/Sinoport/docs/platform-and-station-prd.md)
- 当前前端实现依据：`admin-console` 代码库

本文件的作用不是重复 PRD，而是把 PRD 转成一份可以直接排期、拆任务、分工、联调、验收的开发基线。

本文件覆盖范围：

- 平台管理后台桌面端
- 货站后台桌面端
- PDA 作业终端移动端
- 前端共享组件、状态字典、路由与信息架构
- 与任务编排、单证链、审计、接口治理相关的联调任务

本文件不覆盖：

- 财务结算、合同、发票、利润报表
- 客户门户与外部查询门户
- 真实链上写入与资产化能力
- 与当前升级主线无关的框架迁移
- 任何后台服务、数据库、对象存储、消息队列、真实权限中心、真实工作流引擎的实现代码

## 1.1 当前阶段红线

当前阶段只做前端 demo，不做后台实现。

这条红线必须在所有开发任务中保持一致：

1. 不新增任何后端项目、服务端路由、数据库模型、持久化逻辑。
2. 不实现真实登录鉴权、真实文件上传、真实任务编排、真实审计入库、真实接口同步。
3. 所有“接口”“导入”“放行”“编排”“审计”“权限”能力，当前阶段只做前端原型表达。
4. 所有业务数据都继续使用演示数据、前端 mock 适配层或本地状态模拟，不以联通真实后端为目标。
5. 当前阶段的目标是把产品结构、页面边界、状态机表达、任务流表达和现场交互原型做完整。
6. 到下一阶段才开始基于本轮前端 demo 去落真实后端能力。

## 2. 现有前端代码基线与必须遵守的规范

### 2.1 技术基线

当前前端不是 Mantine 技术栈，而是基于 `Mantis` 管理后台模板的如下组合：

- `React 19`
- `Vite 7`
- `MUI 7`
- `React Router 7`
- `Axios + SWR`
- `Mantis` 的布局、主题、卡片与导航体系
- 移动端原型中使用 `useLocalStorage` 做现场状态暂存

### 2.2 本轮升级必须遵守的开发规范

1. 不引入新的 UI 框架。本轮继续沿用 `MUI + Mantis`，不切换到 `Mantine`，也不混入第二套设计系统。
2. 不做大规模框架迁移。本轮不引入 Redux、React Query、TypeScript 全量迁移等高成本改造。
3. 桌面端页面继续遵循现有骨架：`Grid container` + `PageHeader` + `MetricCard/MainCard` + 表格/列表/表单区块。
4. 移动端页面继续遵循现有模式：路由壳页面尽量薄，业务逻辑沉到 `*-shared.jsx`，底部固定任务导航保留。
5. 路由、侧边菜单、顶部搜索、工作区入口必须同步维护，不能只加页面不加导航。
6. 新状态、新标签、新口径必须进入统一字典，不能散落在各页面硬编码。
7. 通用业务组件优先补在 `components/sinoport`，避免每个页面各自重复实现卡片、状态块、对象摘要块。
8. 演示数据仍可阶段性保留，但新的页面数据接入必须优先设计“领域适配层”，不能把 API 拼装逻辑直接塞进页面。
9. `useLocalStorage` 仅用于 PDA 原型阶段或离线兜底，正式流程必须为服务端持久化预留切换位。
10. 现有主题、断点、排版和 `MainCard` 风格视为固定基线，新增页面必须视觉一致。
11. 当前阶段所有“接口接入”都只做前端 contract placeholder，不产生任何真实后端实现代码。

### 2.3 当前代码中已形成的可复用约束

| 约束项 | 当前实现 | 本轮要求 |
| --- | --- | --- |
| 页面入口 | `src/pages/platform`、`src/pages/station`、`src/pages/mobile` | 新页面继续按领域归档，不新增平行散目录 |
| 路由注册 | `src/routes/MainRoutes.jsx`、`src/routes/MobileRoutes.jsx` | 每个新模块必须同步补路由 |

## 4. 升级项目总体策略

### 4.1 核心策略

1. 先修正信息架构，再持续开发页面。否则旧菜单会持续拖累模块理解。
2. 先统一对象模型、状态字典、任务口径，再接入更多页面。否则页面只会重复造概念。
3. 先做 `MME` 样板站的进港全链路，再复制到其它站点与出港场景。
4. 前端优先搭好“对象详情 + 任务视图 + 文件视图 + 审计视图”四套共通骨架，再铺业务。
5. 移动端不另造一套系统，继续复用当前共享页模式，把节点任务抽象出来。

### 4.2 本轮建议的信息架构升级

| 当前模块 | PRD v1.0 目标模块 | 动作 |
| --- | --- | --- |
| 平台默认入口：货站管理 | 运行态势中心 | 新增页面并改默认入口 |
| 货站管理 | 货站与资源管理 | 原页面升级 |
| 规则中心 | 规则与指令引擎 | 原页面升级 |
| 文件中心 | 单证与指令中心 | 原页面重命名并扩展 |
| 提单管理 | 提单与履约链路 | 从进/出港中抽离为统一模块 |
| 手机理货 | PDA 作业终端 | 统一命名，保留原型路径但外显重命名 |
| 无 | 作业指令中心 | 新增 |
| 无 | 班组 / 区位 / 设备管理 | 新增 |
| 无 | 主数据与接口治理 | 新增 |

## 8. 按模块进一步细化的建议

### 8.1 平台运行态势中心

建议页面结构：

- 顶部：全网 KPI、今日阻塞链路、样板站 SLA 达成率
- 中部左侧：站点健康度矩阵
- 中部右侧：待处理异常、待审批变更、接口异常
- 下部：链路态势时间线、规则命中告警、事件回溯入口

### 8.2 提单与履约链路

建议不要继续按“进港提单”和“出港提单”分裂，而是统一成一套对象模型：

- 列表视角按 `Shipment/AWB`
- 详情页展示上下游航班、相关 ULD/PMC、转运卡车、NOA/POD、异常、任务、文件
- 详情页必须能反向跳到航班、文件、任务、异常页面

### 8.3 单证与指令中心

建议拆为三个标签页：

- 文件台账
- 指令模板
- 放行与阻断

建议重点能力：

- 文件版本覆盖
- 预览与回连
- 文件驱动的状态放行
- 指令模板与作业任务的绑定关系

### 8.4 作业指令中心

建议视图最少包含：

- 我的任务
- 站内任务池
- 待升级任务
- 待复核任务
- 已完成任务

每个任务都应直接暴露：

- 所属节点
- 责任角色
- SLA
- 当前状态
- 需要的证据
- 上游阻断原因
- 下游触发结果

### 8.5 PDA 统一任务卡

统一任务卡建议固定包含：

- 任务标题
- 节点与角色
- 关联对象
- SLA 与优先级
- 必填字段
- 必传证据
- 异常上报入口
- 完成前校验提示

当前已实现的拆板、计数、打托、装车、收货、集装器等原型，后续都应该被收敛到这一个任务卡模型下面。

## 9. 项目治理规则

1. 任何新增页面都必须同步更新路由、菜单、顶部搜索和工作区入口。
2. 任何新增状态都必须同步更新状态字典和 `StatusChip` 口径。
3. 任何新增主对象都必须先进入统一对象字典，再进入页面开发。
4. 任何文件相关功能都必须明确“它驱动什么状态、放行什么任务、阻断什么动作”。
5. 任何 PDA 页面都必须先判断能否复用统一任务卡，不得直接再写一套孤立流程页。
6. 任何跨模块页面都必须支持对象回连，避免形成新的信息孤岛。
7. 任何提测都必须附带本模块的验收清单和 PRD 对应章节。
8. 当前阶段禁止以“顺手补一下后端”为由加入任何服务端实现代码。
9. 任何涉及真实接口的页面，都必须先用 mock 数据和前端状态机完成 demo，再等下一阶段接后端。

## 10. 最终结论

当前代码库已经有足够好的壳层，可以直接承接 v1.0 升级，不需要推倒重做。真正要做的是三件事：

- 把旧版 v0.2 的信息架构升级到 v1.0 的模块边界
- 把“页面原型”升级为“对象 + 状态 + 任务 + 文件驱动”的系统前端
- 以 `MME` 样板站为核心，把桌面端与 PDA 端先闭环，再复制到多站点

后续开发建议严格以本文件为排期基线，以 PRD 为业务依据，以当前 `admin-console` 代码结构为实现约束。

补充约束：

- 当前阶段的交付物是高保真前端 demo，不是可上线的全栈系统。
- 下一阶段再基于本轮前端原型去拆真实后端、接口和持久化实现。
