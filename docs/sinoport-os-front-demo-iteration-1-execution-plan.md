# Sinoport OS 第一批前端 Demo 执行排期与分工

> 历史文档说明：本文件为第一批执行计划，当前模块正式命名已在后续 PRD 与任务文档中更新。

## 1. 文档定位

本文件是第一批前端 demo 任务的执行层文档。

它不是新的规划稿，而是把以下文档转成可以直接分配给开发的执行表：

- 总规划：[sinoport-os-upgrade-task-plan.md](/Users/lijun/Downloads/Sinoport/docs/sinoport-os-upgrade-task-plan.md)
- 第一批清单：[sinoport-os-front-demo-iteration-1.md](/Users/lijun/Downloads/Sinoport/docs/sinoport-os-front-demo-iteration-1.md)

唯一前提：

- 当前阶段只做前端 demo，不做任何后台实现代码。

## 2. 默认团队模型

默认按 `4 名前端 + 1 名产品/验收协同` 组织。

如果你们当前人数不足 4 人，可以按本文最后的压缩方案执行。

### 2.1 角色分工

| 角色 | 负责范围 | 主要产出 |
| --- | --- | --- |
| FE-1 基础架构 | 路由、菜单、搜索、状态字典、对象模型、mock 适配层、通用组件、平台页 | 第一批的底层基线 |
| FE-2 平台与货站总览 | 平台运行态势中心、规则与指令引擎、主数据与接口治理、货站看板 | 平台主控和桌面首页 |
| FE-3 货站业务桌面 | 进港、出港、提单与履约链路、单证与指令中心、作业指令中心 | 货站桌面业务主干 |
| FE-4 PDA | 统一任务卡、通用动作层、`MME` 样板站进港任务流 | 移动端 demo 主干 |
| 产品/验收协同 | PRD 对齐、页面验收、演示脚本、场景核对 | 保证 demo 口径统一 |

### 2.2 写入范围建议

为了减少冲突，第一批建议按以下文件范围管理所有权：

| 角色 | 主写文件范围 |
| --- | --- |
| FE-1 | `admin-console/src/routes/*`、`admin-console/src/menu-items/*`、`admin-console/src/layout/Dashboard/Header/HeaderContent/data/*`、`admin-console/src/components/sinoport/*`、`admin-console/src/data/*` |
| FE-2 | `admin-console/src/pages/platform/*`、`admin-console/src/pages/station/dashboard.jsx` |
| FE-3 | `admin-console/src/pages/station/inbound*`、`admin-console/src/pages/station/outbound*`、新增 `station/shipments*`、`station/documents*`、`station/tasks*` |
| FE-4 | `admin-console/src/pages/mobile/*`、新增 `admin-console/src/components/sinoport/mobile/*` |

说明：

1. `data` 和 `components/sinoport` 由 FE-1 主持，其他人通过约定字段接入。
2. FE-3 与 FE-4 都依赖统一对象和状态字典，因此第一周不能跳过 FE-1 的基线工作。
3. 涉及菜单、搜索、workspace 的改动，默认由 FE-1 统一收口。

## 3. 第一批执行目标

本批次的成功标准不是页面数量，而是形成一个能稳定演示的主干：

1. 平台端已经是 v1.0 信息架构，不再停留在 v0.2 菜单。
2. 货站桌面端已经围绕“对象 + 状态 + 任务 + 文件”组织，不再只是静态列表。
3. PDA 已经具备统一任务卡和统一动作层。
4. `MME` 样板站可以演示一条完整进港链路。
5. 所有演示都依赖 mock 数据和前端状态，不依赖真实后端。

## 8. 验收维度

第一批的验收不看后端联通，只看 demo 是否讲清楚产品逻辑。

### 8.1 平台端

- 能否看懂全网态势
- 能否看懂规则如何驱动任务
- 能否看懂主数据和接口治理口径

### 8.2 货站端

- 能否看懂进港/出港状态机
- 能否看懂提单与履约链路
- 能否看懂文件、任务、异常之间的关系

### 8.3 PDA 端

- 能否围绕统一任务卡完成现场操作演示
- 能否看懂阻断原因和下一步动作
- 能否完成 `MME` 样板链路

## 9. 风险与控制

| 风险 | 表现 | 控制方式 |
| --- | --- | --- |
| 基线未冻结就并行开发 | 字段、状态、命名反复改 | 第一周必须冻结 A-01 到 A-04 |
| 页面先做、对象后补 | 各页字段不一致 | 所有业务页先依赖统一对象模型 |
| PDA 继续沿旧页面演化 | 任务卡模型无法统一 | 先做统一任务卡，再改造链路 |
| 单证与指令中心继续只做台账 | 无法体现 PRD 主线 | 第四周必须补放行与阻断表达 |
| 团队顺手做后端 | 范围失控 | 所有任务说明明确禁止后端实现 |

## 10. 人数不足时的压缩方案

### 10.1 如果只有 3 名前端

按以下方式合并角色：

| 角色 | 合并后范围 |
| --- | --- |
| FE-A | FE-1 + FE-2 |
| FE-B | FE-3 |
| FE-C | FE-4 + 验收收口 |

建议排期从 5 周拉到 6-7 周。

### 10.2 如果只有 2 名前端

按以下方式执行：

| 角色 | 范围 |
| --- | --- |
| FE-A | 基线、平台、货站桌面 |
| FE-B | PDA、文件放行、阻断、验收清单 |

建议把第一批范围收缩为：

- 平台运行态势中心
- 货站看板
- 进港主链路
- 单证与指令中心
- 作业指令中心
- `MME` PDA 进港任务流

出港桌面升级和主数据治理页可顺延到第二批。

## 11. 结论

第一批执行时，最重要的是保证并行不打架、页面口径不漂移、demo 逻辑不断裂。

如果严格按这份分工和周排期执行，第一批结束时就会得到一套可稳定演示的 v1.0 前端 demo 主干，而不是一组彼此松散的新页面。
